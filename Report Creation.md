# Report Creation

## Review Existing Reports and Creation Tools

**Activities**
- Identify the reporting tools currently in use and document whether these will continue to be used, be replaced, or run alongside the new solution.
- Identify the number of users who create their own reports and document the tools and interfaces they currently use.
- Identify any known pain points in the current report creation experience that the new solution should address.

**Outcomes**
- Documented reporting tools currently in use and their expected status in the new solution.
- Documented number of report creators and the tools and interfaces they currently use.
- Documented known pain points in the current report creation experience.

**Pre-requisites**
- A stakeholder with knowledge of current report creation practices is available for a workshop.

**Basis of Estimate**
- Number of reporting tools currently in use
- Number of report creators to consider
- Extent of known pain points to document

## Capture Report Creation Requirements

**Activities**
- Capture whether self-service reporting is a requirement, and document the customer's expectations around what report creators will be able to access and build against.
- Identify whether a certified or endorsed dataset model is required to ensure report creators are working from a consistent, governed data source.
- Capture whether any corporate branding, formatting, or design standards apply to reports, and identify whether templates will be required.
- Capture whether any reports will need to be delivered on a scheduled basis, and identify the recipients, format, and frequency expected.

**Outcomes**
- Documented self-service reporting requirements, including what report creators are expected to access and build against.
- Documented requirements for a certified or endorsed dataset model.
- Documented corporate branding, formatting, or design standards applicable to reports, and whether templates will be required.
- Documented scheduled report delivery requirements, including recipients, format, and frequency.

**Pre-requisites**
- Outputs from the Review Existing Reports and Creation Tools task are available.
- Named stakeholders with knowledge of report creation expectations are available for a workshop.

**Basis of Estimate**
- Complexity of self-service requirements
- Number of reports requiring scheduled delivery
- Whether branding or template requirements need investigation

## Capture Report Governance and Standards

**Activities**
- Capture the expected report refresh frequency and data latency requirements.
- Capture whether any existing report development standards should be carried forward, and identify any new standards that need to be established as part of the project.
- Capture the process for promoting a self-service report into a governed, organisation-wide report, and identify who holds approval authority.
- Capture who will be responsible for maintaining reports after delivery, and identify whether this sits with the customer, a specific team, or individual report owners.

**Outcomes**
- Documented expected report refresh frequency and data latency requirements.
- Documented existing report development standards to carry forward, and any new standards to be established.
- Documented process for promoting self-service reports to governed, organisation-wide status, including approval authority.
- Documented post-delivery report maintenance responsibilities.

**Pre-requisites**
- Outputs from the Capture Report Creation Requirements task are available.

**Basis of Estimate**
- Complexity of existing or new report development standards
- Number of governance steps required for the promotion process
- Number of stakeholders involved in report maintenance
