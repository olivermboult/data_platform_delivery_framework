# Report Design

## Capture Report Design Requirements and Context

**Activities**
- Identify which reports should be prioritised for design, and document their relative order.
- Capture the intended audience for each priority report and what they need to be able to do with it.
- Identify the key questions each report must answer for its intended audience.
- Capture the level of detail required at each layer of each report, from executive summary through to operational drill-through where applicable.
- Identify who holds sign-off authority for report designs.

**Outcomes**
- Documented list of reports prioritised for design, in agreed order.
- Documented intended audience for each priority report and their expected tasks and decisions.
- Documented key questions each report must answer for its intended audience.
- Documented level of detail required at each layer of each report.
- Identified sign-off authority for report designs.

**Pre-requisites**
- Outputs from the Capture Report Creation Requirements task are available.
- Named business stakeholders for each priority report are available for a workshop.

**Basis of Estimate**
- Number of priority reports requiring design
- Complexity of layered detail requirements
- Number of stakeholders required per report

## Capture Visual and Structural Requirements

**Activities**
- Identify any existing reports, dashboards, or mockups the business considers a useful reference point, whether to replicate or improve upon.
- Capture the branding and design standards that apply to the reports, and document where these are defined and who owns them.
- Identify any interactivity requirements, such as filtering, drill-through, or dynamic parameters, that must be reflected in the report designs.
- Capture whether reports will need to be accessible on mobile devices or embedded within other platforms, and identify any constraints this creates for the layout or design.
- Identify any data elements likely to be required in reports that may be unavailable or of uncertain quality, and document these as risks.

**Outcomes**
- Documented existing reports, dashboards, or mockups identified as reference points for the new designs.
- Documented branding and design standards applicable to reports, including where they are defined and who owns them.
- Documented interactivity requirements, such as filtering, drill-through, and dynamic parameters.
- Documented mobile access or embedding requirements and any constraints they create for layout or design.
- Documented data elements identified as potentially unavailable or of uncertain quality, recorded as risks.

**Pre-requisites**
- Outputs from the Capture Report Design Requirements and Context task are available.

**Basis of Estimate**
- Number of priority reports in scope
- Complexity of interactivity requirements
- Extent of mobile or embedding requirements
