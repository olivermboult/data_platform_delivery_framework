# Report Audience & Distribution

## Identify Report Users and Access Requirements

**Activities**
- Identify the target user groups for each report, covering whether consumers are internal staff, external stakeholders, or both, and document the estimated number of users in each group.
- Identify whether different user groups require different levels of access to report data, and document any access tier requirements.
- Review current report usage patterns in the existing solution, identifying which reports are heavily used, which are redundant, and which are missing.

**Outcomes**
- Documented target user groups for each report, including whether consumers are internal or external and estimated user numbers.
- Documented access tier requirements for each user group.
- Documented current report usage patterns, including heavily used, redundant, and missing reports.

**Pre-requisites**
- A stakeholder with knowledge of existing report usage and user groups is available for a workshop.

**Basis of Estimate**
- Number of distinct user groups
- Availability of existing usage data or analytics
- Number of reports in scope for usage review

---

## Capture Report Distribution Requirements

**Activities**
- Identify the distribution mechanisms required for each report, such as published apps, email subscriptions, embedded reports, external sharing, or mobile access.
- Identify whether any reports need to be accessible to users outside the organisation, and document any constraints this creates.

**Outcomes**
- Documented distribution mechanisms required for each report.
- Documented external access requirements and any constraints they create.

**Pre-requisites**
- Outputs from the Identify Report Users and Access Requirements task are available.

**Basis of Estimate**
- Number of reports with distinct distribution requirements
- Whether external access is in scope

---

## Capture Adoption and Discoverability Requirements

**Activities**
- Identify current adoption challenges and barriers within the existing reporting environment, and capture what the new solution should address.
- Capture how users will discover and navigate to reports, and identify whether a centralised app, landing page, or catalogue is expected.
- Capture whether training or onboarding materials will be required to support user adoption of the new solution.
- Capture whether usage analytics are required to allow the business to track adoption and identify underused content.

**Outcomes**
- Documented current adoption challenges and barriers in the existing reporting environment, and what the new solution should address.
- Documented approach to report discovery and navigation, including any requirements for a centralised app, landing page, or catalogue.
- Documented training or onboarding material requirements for user adoption.
- Documented usage analytics requirements.

**Pre-requisites**
- Outputs from the Identify Report Users and Access Requirements task are available.

**Basis of Estimate**
- Number of distinct user groups requiring onboarding support
- Complexity of navigation and discoverability requirements

---

## Capture Report Lifecycle Requirements

**Activities**
- Capture the process for requesting changes to reports after delivery, and identify who is responsible for managing and prioritising these requests.
- Capture the governance process for publishing reports to shared or organisation-wide workspaces, and identify who holds approval authority.

**Outcomes**
- Documented process for requesting and managing post-delivery report changes, including identified responsible parties.
- Documented governance process for publishing reports to shared or organisation-wide workspaces, including approval authority.

**Pre-requisites**
- Outputs from the Capture Adoption and Discoverability Requirements task are available.

**Basis of Estimate**
- Complexity of post-delivery change management requirements
- Number of governance steps required for the publishing process
